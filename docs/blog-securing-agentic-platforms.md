# Securing an Agentic Platform with Consul, Vault, and SPIFFE

> A field-tested pattern for running MCP-driven AI agents on GKE behind a
> service mesh, with HCP Vault Dedicated as the identity and credential
> broker.

This post walks through what
[`consul-mcp-agents`](https://github.com/hashicorp/consul-mcp-agents) does,
why it does it that way, and how the resulting platform maps onto the OWASP
Top 10 for LLM applications. It assumes you've shipped microservices on
Kubernetes before; everything else — SPIFFE, MCP, terminating gateways,
agent RBAC — we'll build up from first principles.

---

## 1. What this repo is for

The repo deploys an end-to-end **agentic AI platform** on Google Kubernetes
Engine:

- **MCP (Model Context Protocol) servers** that expose Google Cloud
  primitives (GCS, BigQuery, Compute Engine) as LLM-callable tools.
- An **MCP-client agent** (Anthropic or OpenAI SDK, no LangChain) that lets
  a human user chat with those tools via a browser-based terminal (`ttyd`).
- A **Consul Dataplane** service mesh wired through every pod, so that
  agent → server traffic is mTLS-authenticated by SPIFFE identity.
- An **HCP Vault Dedicated** cluster acting as the platform's identity
  fabric — root CA for the mesh, dynamic credential broker for Google Cloud,
  and central store for the few static secrets the platform needs (LLM API
  keys, app config, capability policies).

The thesis is straightforward: an agent calling an MCP server is, from a
security architecture point of view, just another microservice call. The
caller has an identity, the callee has an identity, and a policy engine has
to decide whether the call is allowed. Where it gets interesting is that
**the caller's "intent" is generated by an LLM at runtime**, so the policy
engine has to be defended at multiple layers — service-to-service, tool
selection, and per-API guardrails.

### Why let Kubernetes assign SPIFFE SVIDs

[SPIFFE](https://spiffe.io/) is a spec for service identities — it gives
every workload a globally unique URI like
`spiffe://example.consul/ns/mcp-agents/dc/dc1/svc/mcp-data-server`,
embedded as a `URI SAN` in an X.509 cert. Anyone presenting that cert in an
mTLS handshake can be authenticated as that workload, by anyone else who
trusts the SPIFFE root.

There are two ways to get SPIFFE SVIDs into your pods:

1. **Run [SPIRE](https://github.com/spiffe/spire) yourself.** Deploy a SPIRE
   server, a SPIRE agent DaemonSet, write workload registration entries
   that map (Kubernetes namespace, ServiceAccount, container image) to
   SPIFFE IDs, plumb the [Workload API](https://github.com/spiffe/spiffe/blob/main/standards/SPIFFE_Workload_API.md)
   socket into every pod, and then teach your application code to fetch
   SVIDs from it.
2. **Let Consul do it.** Consul Connect is SPIFFE-compliant. The
   `connect-inject` admission webhook adds an Envoy sidecar to your pods,
   and Consul mints a SPIFFE-shaped X.509 SVID for that sidecar
   automatically — no SPIRE registration entries, no Workload-API socket,
   no application changes. Pods stay vanilla.

The second approach is what `consul-mcp-agents` uses, with one twist: the
SPIFFE root isn't Consul's built-in CA, it's
**Vault PKI** (`pki-consul/connect-root` mount). Consul's `connect.ca_provider`
is set to `vault`, so every leaf SVID issued in the mesh chains back to a
Vault-managed root.

The benefit isn't theoretical. SPIRE registration is a real operational
load — you write entries when you deploy, you reconcile entries when pod
identities drift, you debug entries when handshakes fail. Consul Connect
collapses all of that onto annotations:

```yaml
metadata:
  annotations:
    consul.hashicorp.com/connect-inject: "true"
    consul.hashicorp.com/connect-service-upstreams: "mcp-data-server:20000,mcp-compute-server:20001"
```

That's it. The pod gets an SVID, an Envoy sidecar, and an mTLS-protected
upstream connection on `localhost:20000`. The application code dials a
plain `localhost` socket — no TLS termination, no SDK, no registration
file. **The mesh layer owns identity end to end.**

### Why terminating gateways

A SPIFFE trust domain is a closed world: if a workload doesn't carry an
SVID under your trust domain's URI scheme, the mesh can't authenticate it.
That's a feature for service-to-service traffic, but it's a problem the
moment a service has to call something outside the mesh — a public Google
API, an HCP Vault endpoint, an Anthropic or OpenAI completion API.

You have two unattractive options out of the box:

- **Bypass the mesh.** Let pods open egress sockets directly. Now the egress
  hop is unauthenticated from the mesh's point of view, and you've lost the
  audit trail for what `mcp-data-server` is actually talking to.
- **Re-do mTLS at the application layer.** Have the MCP server speak a
  separate TLS stack to GCP. Now you maintain two trust models per service.

Consul's
[**terminating gateway**](https://developer.hashicorp.com/consul/docs/connect/gateways/terminating-gateway)
solves this without compromise. A terminating gateway is itself a mesh
service — it has an SVID, it accepts mTLS connections from upstream
sidecars under a Connect intention — but its egress side speaks ordinary
TLS to a registered external destination. So the data flow becomes:

```
mcp-data-server (Envoy + SVID)
      │ mTLS, intention-gated
      ▼
terminating-gateway (Envoy + SVID)
      │ TLS / TLS-SNI
      ▼
storage.googleapis.com  /  bigquery.googleapis.com  /  api.anthropic.com
```

From the mesh's point of view, every hop a service makes is observable and
authorisable. The trust domain effectively extends to "the gateway plus
whatever it's configured to talk to," and that configuration is declarative
Consul state, not application code. That's the right place for it.

> The repo currently registers external services as Consul `service` entries
> ready to be fronted by a terminating gateway; wiring the gateway itself
> into the GKE Helm release is the next iteration. The pattern is what
> matters: external dependencies belong on the mesh's terms, not below it.

### Microservices and agentic platforms have the same problem space

Stripped down, a microservices platform asks three questions on every call:

1. **Who is calling?** (service identity)
2. **Are they allowed to make this call?** (authorization policy)
3. **What credentials should the callee use to do its work?** (secret
   delivery)

An agentic platform asks the same three questions, with one added wrinkle:
**the call itself is chosen by an LLM at runtime**. So:

- *Who is calling* still needs SPIFFE-style cryptographic identity — the
  agent pod must prove to the MCP server it's the agent pod.
- *Are they allowed* now has two layers: the service-to-service layer
  (Consul intentions), *and* the tool-selection layer (does this user's
  role allow this tool, with this argument shape?).
- *What credentials should the callee use* must produce short-lived,
  scoped credentials, because LLMs sometimes do unexpected things and you
  want the blast radius of "unexpected" capped at five minutes.

That third point is why Vault is non-negotiable. A long-lived GCP service
account key inside an MCP server is a footgun. A 5-minute impersonation
token, brokered freshly per call, isn't.

## 2. OWASP Top 10 for LLM Applications: what this design mitigates

The [OWASP Top 10 for LLM
Applications](https://owasp.org/www-project-top-10-for-large-language-model-applications/)
catalogues the most common ways LLM-backed systems fall over. Below is how
the repo's design lines up against each, with a frank note where it
doesn't.

| OWASP item | What it means | What this repo does |
|---|---|---|
| **LLM01 — Prompt Injection** | A malicious prompt or tool output makes the model behave outside spec. | Tool outputs are sanitised before reaching the model: errors collapse to `type(exc).__name__`, full details stay server-side. The model never sees raw stack traces or DB error strings. |
| **LLM02 — Insecure Output Handling** | The application blindly trusts model output as code/SQL/shell. | BigQuery queries are parsed with `sqlglot` AST and rejected if they contain DML/DDL — *no regex bypass*. Compute machine types are checked against an `ALLOWED_MACHINE_TYPES` whitelist before any GCE call. |
| **LLM03 — Training Data Poisoning** | Out of scope; we don't train. |
| **LLM04 — Model Denial of Service** | An attacker makes the model burn unbounded resources. | BigQuery jobs cap at `BQ_MAX_BYTES_BILLED` (1 GiB default). Compute scope is whitelist-only. The agent CLI rate-limits per user session. |
| **LLM05 — Supply Chain Vulnerabilities** | Compromised base images, dependencies, or downloaded binaries. | Container images are built on Chainguard Wolfi (`cgr.dev/chainguard/python:latest-dev`) — near-zero CVE surface vs. Debian-based bases. Vault binary is GPG-signature-verified against HashiCorp's published `SHA256SUMS`. ttyd is checksum-verified against the upstream GitHub release. Python deps are upper-bound-pinned in `pyproject.toml`. |
| **LLM06 — Sensitive Information Disclosure** | Secrets leak through model output, logs, or kubectl. | Secrets never live in K8s manifests or container env vars at rest. LLM keys come from Vault → vault-agent → tmpfs `emptyDir` (memory only). GCP credentials are 5-minute impersonation tokens. The ttyd password is plumbed via `secret_key_ref`, so `kubectl describe pod` never prints it. |
| **LLM07 — Insecure Plugin Design** | Plugins (= MCP servers here) trust their callers and grant broad capability. | Each MCP server runs with its own Kubernetes ServiceAccount, its own Vault role (`mcp-data-server`, `mcp-compute-server`), its own GCP impersonation target with **least-privilege IAM** (`storage.objectAdmin` not `storage.admin`; `compute.instanceAdmin.v1` not `compute.admin`). Consul intentions enforce that only the agent pod can reach a server. |
| **LLM08 — Excessive Agency** | The agent has more permissions than the task requires. | Two-layer RBAC: Vault policies cap what the pod can read; `capabilities.yaml` filters which tools the LLM is allowed to *select* per user role (`operator` / `analyst` / `viewer`). Filtering happens *before* the tool list is sent to the model — the model can't ask for what it can't see. |
| **LLM09 — Overreliance** | Treating model output as ground truth. | Largely a UX concern; out of scope for the platform layer, though the read-only-query default for `viewer` role is a deliberate hedge. |
| **LLM10 — Model Theft** | Out of scope; we don't host models. |

The newer
[**OWASP Agentic AI Top 10**](https://owasp.org/www-project-agentic-ai/) adds
some agent-specific risks worth calling out:

- **AI2 — Tool Misuse / AI3 — Privilege Compromise:** mitigated by the same
  least-privilege impersonation chain — there is no shared, broad GCP
  credential, so a compromised tool selection still can't write to a
  bucket the data agent doesn't have `objectAdmin` on.
- **AI4 — Resource Overload:** BigQuery byte cap, machine-type whitelist.
- **AI8 — Repudiation & Untraceability:** every Vault read is in HCP Vault
  audit logs (org-managed, immutable); every Consul intention check is in
  the dataplane access log; every BigQuery and GCS call is in GCP audit logs
  attributed to the impersonated SA, not a shared identity.

What this design **doesn't** mitigate, and why it's honest to say so:

- **Prompt injection from data ingested by tools** (e.g., a BigQuery row
  containing `Ignore previous instructions…` getting echoed back to the
  agent). The repo doesn't do output-side prompt filtering; that's a
  product decision more than an infra one.
- **Repudiation at the LLM layer.** GCP audit logs say "data-agent SA read
  this object." They don't say which user's session caused that read.
  Closing that loop requires propagating user identity into the MCP call
  chain — work in progress.

## 3. The deployment workflow

The repo is operated through a single `Taskfile.yml` that wraps Terraform,
Packer, gcloud, kubectl, helm, and a fistful of bash scripts. The whole
thing comes up with `task all`, but it's worth understanding the phases
because the dependency graph isn't linear.

```
preflight              ── tooling + GCP auth check
  │
ingress:sync-ip        ── pin allowed_ingress_cidrs to my current IP/32
  │
token:ensure           ── generate Consul bootstrap token if absent
  │
tf:backend:init        ── create GCS state bucket, init backend
  │
packer:build           ── bake Consul-server VM image with vault-agent
  │
phase1:apply           ── network → HCP Vault → Vault PKI → Vault config → Consul VMs
  │
phase2:apply           ── GKE cluster → Consul Helm (TLS via Vault PKI CA)
  │
phase3:apply           ── Vault K8s auth → Docker build/push → MCP deployments
  │
phase4:apply           ── full reconcile (catches any drift the targeted applies missed)
  │
summary                ── print URLs, credentials hint, smoke-test command
```

The phases exist because some Terraform inputs aren't knowable until
earlier resources are created. Specifically:

- **HCP Vault** must exist before `vault_pki` and `vault_config` can write
  any policies or mounts — they target the cluster URL, which is a Vault
  module output.
- **Vault PKI** must exist before the Consul VMs boot, because the
  `vault-agent` running on each VM authenticates via GCP IAM and pulls its
  TLS material from `pki-consul/issue/consul-server` on first boot. No PKI,
  no certs, no Consul start.
- **GKE** must exist before the `module.gke.kubernetes_*` resources can
  resolve, and before Vault's `auth/kubernetes/config` can be filled in
  with the cluster's private endpoint and CA.
- **Consul Helm chart** must finish (and its `consul-server-acl-init` Job
  must complete) before MCP pods can `connect-inject` — otherwise the
  injection webhook fails closed.

The phase split also makes the platform **resumable**: a failure in
phase 2 doesn't drop everything you built in phase 1. The auto-detection of
GKE readiness in `task gke:ensure-ready` writes `gke_cluster_ready = true`
back into `tfvars` automatically once the cluster is up, so the next run
short-circuits past resources that already exist.

## 4. Component breakdown — and where Vault feeds each one

This is the actual architecture, walked top to bottom. Pay attention to the
"where Vault feeds it" column in your head — it's the through-line.

### HCP Vault Dedicated

The identity fabric. It runs three secret engines, two auth methods, and
one PKI mount that matters most:

- **`pki-consul/`** — root + intermediate. Root TTL 10y, intermediate 5y,
  leaf 72h. This is the SPIFFE root for the whole Consul mesh. It also
  issues server-TLS certs for the Consul VMs themselves (RPC, HTTPS API,
  gossip). `prevent_destroy = true` — a destroy here invalidates every
  mTLS cert in the mesh.
- **`gcp/`** — Google Cloud Secrets Engine. Configured with two
  *impersonated-account* roles, `data-agent-gcp` and `compute-agent-gcp`,
  each pointing at a least-privilege GCP service account. Reads here mint
  fresh 5-minute OAuth2 access tokens via GCP's `generateAccessToken` API.
- **`secret/`** — KV v2. Holds `mcp-agents/config` (a single
  `settings_yaml` field with the rendered app config), `mcp-agents/policies`
  (the capabilities/RBAC YAML), `mcp-agents/llm-keys` (Anthropic + OpenAI
  API keys), and `consul/acl-token` (the bootstrap token, written
  post-bootstrap).
- **`auth/gcp`** — IAM auth method. The Consul VMs authenticate via their
  GCE instance-metadata-signed JWTs.
- **`auth/kubernetes`** — K8s auth method. Pods in the GKE cluster
  authenticate via their projected SA tokens, validated through a dedicated
  `vault-reviewer` SA with `system:auth-delegator`.
- **`auth/userpass`** — three operator accounts (`alice`, `bob`, `carol`)
  with different policies, used for the demo of capability-driven RBAC.

### Network

A standard GCP VPC with private subnets, Cloud NAT for egress, an HCP HVN
peered into the VPC for Vault, and `allowed_ingress_cidrs` validation that
**rejects `0.0.0.0/0`**. The same CIDR list is reused for the GKE master
authorised networks and the Consul ingress-gateway LoadBalancer
`loadBalancerSourceRanges` — three knobs, one source of truth.

### Consul VMs (GCE)

Three GCE instances built from a Packer-baked AlmaLinux image. The image
ships:

- The Consul binary + a `consul-server.hcl` baseline config with
  `auto_reload_config = true` (so cert rotations hot-reload without SIGHUP).
- A `vault-agent` v1.21.3 binary with a templated config that, on boot,
  authenticates via GCP IAM and renders four files into `/etc/consul.d/`:
  - `tls/server.crt`, `tls/server.key`, `tls/ca-chain.pem` (server TLS)
  - `tls-certs.hcl` (Consul's `tls { defaults { … } }` block pointing at
    the above)
  - `connect-ca.hcl` (the `connect { ca_provider = "vault" … }` block that
    tells Consul to use Vault PKI as its Connect CA)
- A systemd timer running `vault-agent-cert-refresh.sh` every 60h (cert
  TTL 72h) — defence in depth against vault-agent renewal bugs.

**Where Vault feeds it:** `pki-consul/issue/consul-server` (TLS leaf),
`pki-consul/cert/ca` (intermediate for Connect), and the `connect-root` /
`connect-intermediate` paths used by Consul itself when minting Connect
SVIDs.

### GKE cluster

A native `google_container_cluster` (deliberately not the community
module — the cluster is opinionated for Consul Dataplane: Calico network
policy, per-namespace Workload Identity bindings, increased pod density).
STABLE release channel, 45-minute create timeout, optional CMEK via
`gke_database_encryption_key` for application-layer secrets encryption
(CIS 8.5.5).

### Consul Helm release (control plane = the VMs, data plane = the pods)

A standard Consul Helm chart pinned in `gke.tf`, with the external-servers
mode enabled — there's no Consul server StatefulSet, the pods join the VM
cluster. The chart also brings:

- `connect-injector` — the admission webhook that adds Envoy sidecars to
  annotated pods.
- `consul-server-acl-init` Job — bootstraps ACL component tokens. Must
  complete before pods can authenticate, hence `wait_for_jobs = true`.
- An ingress gateway (LB-restricted via the same `allowed_ingress_cidrs`).

### Vault Secrets Operator (VSO) and vault-agent — two patterns, one purpose

For pods, secret delivery uses **vault-agent sidecar/init containers**
rather than VSO CRDs. The pattern is:

- A projected ServiceAccount token mounted at
  `/var/run/secrets/vault/token` (audience `vault`, TTL 2h).
- A vault-agent container (init or sidecar) authenticates to
  `auth/kubernetes` with that token, picks up a Vault token, and renders
  `template` blocks into a shared `emptyDir` with `medium: Memory`.

### MCP **agent** pod (`mcp-agent`)

The user-facing pod. Runs `ttyd` exposing a browser terminal; behind the
terminal is the `vault_mcp_agents` CLI (a thin Anthropic/OpenAI SDK adapter
that drives MCP tool calls — no LangChain). The
`vault-agent-init` container (init only, `exit-after-auth=true`) renders:

| Vault path | Rendered file | Used for |
|---|---|---|
| `secret/data/mcp-agents/llm-keys` | `/vault/secrets/anthropic-key`, `…/openai-key` | LLM API auth |
| `secret/data/mcp-agents/config` | `/vault/secrets/settings.yaml` | App config (Vault addr, GCP project, agent + MCP server URLs) |
| `secret/data/mcp-agents/policies` | `/vault/secrets/capabilities.yaml` | Per-role tool allowlists |

The pod gets an Envoy sidecar with explicit upstream annotations:

```yaml
consul.hashicorp.com/connect-service-upstreams: "mcp-data-server:20000,mcp-compute-server:20001"
```

so the agent process dials `localhost:20000` and `localhost:20001` for SSE
MCP connections — the mesh handles mTLS, identity, and intention checks
between the agent and each server.

**Where Vault feeds it:** static secrets (LLM keys, config, policies) once
at startup, into a tmpfs that never touches node disk.

### MCP **server** pods (`mcp-data-server`, `mcp-compute-server`)

The tool-execution side. Each server runs an SSE-mode MCP transport
(Starlette + uvicorn) on port 8080, exposes a list of tools (GCS,
BigQuery, GCE depending on the server), and answers calls from the agent
pod's Envoy upstream. Each pod has:

- Its own ServiceAccount, scoped to a Vault role of the same name.
- A `vault-agent-init` container *and* a `vault-agent` sidecar
  (`exit_after_auth = false`). The sidecar matters: GCP tokens have a
  5-minute lease and the sidecar's job is to keep `/vault/secrets/gcp-token`
  rewritten in place ahead of every expiry.
- A read-only root filesystem (`read_only_root_filesystem = true`) with a
  tmpfs `emptyDir` at `/tmp` for the readiness sentinel and any
  process-local writes. The vault-agent volume is also tmpfs.

**Where Vault feeds it:** `gcp/impersonated-account/<role>/token` for
ongoing GCP access. The Python server reads `/vault/secrets/gcp-token` at
startup and on auth failure, builds
`google.oauth2.credentials.Credentials(token=…)`, and feeds that into the
GCS / BigQuery / Compute clients. The server contains *no* hvac code, *no*
Vault SDK — vault-agent owns the lease loop, the app reads files.

### Consul intentions

Declarative service-to-service authorisation. The repo registers a small
default-deny set:

- `mcp-agent → mcp-data-server` ALLOW
- `mcp-agent → mcp-compute-server` ALLOW
- `* → *` DENY (implicit when default-deny is configured)

Every other call is rejected at the Envoy sidecar before it reaches the
target service.

## 5. Demoing it

The full deployment takes about 30 minutes the first time (Vault
provisioning is the long pole) and ~10 minutes on subsequent applies. The
flow below assumes you've followed the README's prerequisites and tfvars
setup.

### 5.1. Stand the platform up

```sh
export TF_VAR_hcp_client_secret='…your HCP service-principal secret…'
task all
task summary             # prints the web-terminal LB URL and demo creds
```

### 5.2. Show the SPIFFE identity story

Pull the running cert off any sidecar and look at it:

```sh
kubectl exec -n mcp-agents deploy/mcp-data-server -c consul-dataplane -- \
  cat /consul/connect-inject/leaf-cert.pem | \
  openssl x509 -noout -text | grep -E 'URI|Issuer|Subject:'
```

You'll see an `URI:spiffe://<trust-domain>/ns/mcp-agents/dc/dc1/svc/mcp-data-server`
SAN, an `Issuer` chained back to the Vault PKI intermediate, and a 72-hour
`Not After`. No SPIRE, no registration entry — just an annotation.

### 5.3. Show the vault-agent broker pattern

Two pod types, two patterns:

```sh
# Agent pod — static secrets rendered once
kubectl exec -n mcp-agents deploy/mcp-agent -c vault-agent-init -- \
  echo "init-only, has already exited"
kubectl exec -n mcp-agents deploy/mcp-agent -- ls -la /vault/secrets/
# anthropic-key, openai-key, settings.yaml, capabilities.yaml, .ready

# MCP server — token rotated by the sidecar
kubectl exec -n mcp-agents deploy/mcp-data-server -c mcp-app -- \
  sh -c 'ls -la /vault/secrets/gcp-token && md5sum /vault/secrets/gcp-token'
sleep 60
kubectl exec -n mcp-agents deploy/mcp-data-server -c mcp-app -- \
  md5sum /vault/secrets/gcp-token
# Different hash — the sidecar rewrote it ahead of lease expiry
```

### 5.4. Show capability-driven RBAC at the LLM level

Open the web terminal (`task mcp:url`), log in as **alice** (operator
role) and ask:

```
Create a new e2-medium VM called demo-vm in us-central1-a
```

The agent calls `gce_create_instance`, succeeds.

Now log in as **carol** (viewer role) and ask the same thing. The agent
literally cannot see `gce_create_instance` in its tool list — `capabilities.yaml`
filters the registry before the LLM gets it. The model returns "I don't
have a tool that can do that," not a permission error from GCP. Different
failure mode, much better UX, much smaller blast radius.

### 5.5. Show output handling defences

Still as alice:

```
Run this on BigQuery: DROP TABLE my_dataset.users
```

The agent submits the query, the server parses it with `sqlglot`, sees a
DDL node in the AST, and refuses *before* any GCP call. The model gets a
bounded error string back; it does not retry the query as a different
spelling because the parse rule is structural, not regex.

```
Spin up an n1-megamem-160 VM
```

Same outcome — `ALLOWED_MACHINE_TYPES` whitelist hits before GCE does.

### 5.6. Show the smoke test

```sh
task smoke
```

End-to-end verification across nine sections (Terraform state, AR repo,
HCP Vault, Vault HTTP, Vault auth & engines, Consul VM, GKE, Consul Helm,
MCP pods). PASS / FAIL / SKIP per check, plus a markdown report dropped
under `.runs/smoke-<timestamp>.md` for the demo trail.

### 5.7. Show the audit trail

```sh
# HCP Vault audit logs — every secret read attributed to a Vault token
# whose accessor maps back to a SPIFFE-identified pod.
hcp vault-secrets logs tail --cluster <cluster-id>

# Consul intention denials — try to reach a server you shouldn't.
kubectl run --rm -it --image=curlimages/curl -- \
  curl -k https://mcp-compute-server.mcp-agents.svc:8080
# 503 from the upstream Envoy — intention default-deny in action.
```

## What to take from this

The interesting claim in this repo isn't any single component — Vault,
Consul, and the MCP pattern all stand alone. It's the **composition**:

- Identity is one cryptographic concept (SPIFFE), brokered by the
  infrastructure layer (Consul Connect), rooted in one CA (Vault PKI).
- Authorisation is one declarative model (Consul intentions for service
  edges, capabilities.yaml for tool edges), enforced at multiple layers.
- Secrets are one delivery pattern (vault-agent renders to memory tmpfs),
  whether the consumer is a Consul VM, an agent pod, or an MCP server.
- Privilege is always least and short-lived: 72-hour mesh certs, 5-minute
  GCP tokens, per-role IAM on every impersonation target.

Agentic platforms make the consequences of getting any of those layers
wrong worse, because the caller is non-deterministic. They don't make the
underlying problem new. The microservices security playbook still applies —
you just have to apply all of it, not pick favourites.
